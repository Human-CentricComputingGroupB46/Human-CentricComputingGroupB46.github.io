1.1 The Why

Our group selected the Active track because indoor navigation is a real and frequent friction point in everyday campus movement. In the EB building, users may know a room code such as EB204, but the code alone does not reveal the correct wing, entrance, stair, or corridor route. This creates repeated stopping, backtracking, and uncertainty, especially for first-year students during their first weeks on campus and external visitors who may only enter the building once.

We observed that this problem is not caused by users’ lack of motivation, but by a mismatch between static signage and the moment-to-moment decisions people must make while moving through an unfamiliar building. A student rushing from a previous class may not have time to compare corridor signs, ask staff, or recover from a wrong turn before the next lesson starts.

Therefore, we designed CampusCompass as a point-of-entry decision support tool. Users select where they entered, input a destination room code, and immediately receive a shortest path with plain-language route steps. This supports quick, confident movement under time pressure while reducing cognitive load. The concept is intentionally lightweight, browser-based, and suitable for kiosk deployment, so guidance can be placed where confusion begins rather than requiring every user to install a new app. Through this project, we aim to reduce anxiety, improve first-time wayfinding success, and support more independent campus navigation.

Academic paper

Paper 1
A Generic Approach toward Indoor Navigation and Pathfinding with Robust Marker Tracking

Khan et al., 2019

What it did well
  • Keeps deployment low-cost using only ceiling markers and smartphones, while remaining easy to scale.
  • Enables efficient navigation with automatic shortest-path generation across spaces.
  • Improves first-time use through combined visual and audio guidance.
What it missed
  • Relies on visible markers and stable lighting, limiting reliability.
  • Requires continuous camera use, adding user effort.
  • Lacks low-friction options (e.g., kiosks) and confidence support for stressed users.
Paper 2
INSUS: Indoor Navigation System Using Unity and Smartphone for User Ambulation Assistance

Fajrianti et al., 2023

What it did well
  • Makes navigation intuitive by using AR arrows to guide users along routes.
  • Reduces infrastructure needs by relying on smartphone sensors and QR-based initialization instead of dedicated hardware.
  • Validated in real environments, with prototypes tested in two university buildings and positive user feedback.
What it missed
  • SLAM drift can reduce accuracy, sometimes requiring users to rescan QR codes.
  • Performance is sensitive to how the phone is held and oriented during use.
  • Unity-based 3D system is relatively heavy compared to lightweight, browser-based solutions.
Paper 3
Generating Comfortable Navigable Space for 3D Indoor Navigation Considering Users’ Dimensions

Zhen et al., 2020

What it did well
  • Supports inclusive and accessibility-aware routing decisions for diverse users.
  • Models indoor spaces, obstacles, and functions in a more human-centered 3D representation.
  • Shortest-path routing by incorporating user comfort and body dimensions into navigation planning.
What it missed
  • Focuses more on spatial modeling than on a practical, user-facing navigation interface.
  • Relies on simplified geometric assumptions, which may reduce real-world precision.
  • Does not address key usability needs like fast destination input or beginner-friendly guidance.
Paper 4
NavCog3 in the Wild: Large-Scale Blind Indoor Navigation Assistant with Semantic Features

Sato et al., 2019

What it did well
  • Demonstrates robust large-scale real-world deployment with strong in-the-wild validation.
  • Enhances navigation using semantic cues like doors, landmarks, and nearby points of interest.
  • Provides strong accessibility support with detailed turn-by-turn guidance for blind users.
What it missed
  • Depends on beacon infrastructure and venue-specific installation, limiting portability.
  • Primarily designed for blind navigation rather than general visitor or kiosk-based use.
  • Requires relatively high deployment and maintenance complexity compared to lightweight solutions.

GAP SUMMARY

Research Gap Summary

Existing academic systems often rely on sensors, markers, AR, or large-scale spatial infrastructure. Commercial indoor-map products are powerful, but they are not optimized for a lightweight classroom-code-to-route task in one campus building.

CampusCompass fills this gap by focusing on a simple browser-based flow: room code input → entrance-aware route → step-by-step indoor guidance → arrival confirmation.

Early Research Turned Pain Points into Design Needs

Questionnaire findings were translated into concrete wayfinding requirements for CampusCompass.

1. Data Collection & Target Users

We collected survey responses from 50 questionnaire respondents. The sample included 20 first- and second-year students, 20 third- and fourth-year students, and 10 external visitors. This mix captured both routine classroom-finding problems and first-time navigation difficulties in unfamiliar campus buildings.

2. Questionnaire Evidence

The questionnaire showed that classroom-finding problems were frequent, repeated, and tied to specific building conditions rather than isolated mistakes. We used comparison across response groups and a pain point-to-mechanism mapping matrix to identify which problems appeared most often and which wayfinding supports were most needed.

Problem: Confusing building numbers

20 participants said unclear building numbering led them to the wrong classroom. The strongest matching mechanism is building-specific destination input with clearer room parsing. This suggests students need the system to reduce ambiguity before moving. This supports searchable destination input and route generation based on explicit building context.

Problem: Unclear floor markings

12 participants reported difficulty because floor labels were not clear enough. The most relevant mechanism is floor-aware route guidance with explicit level transitions. This indicates students need confirmation of when to go upstairs, downstairs, or stay on the same level. Therefore, route output should include strong floor cues and step-by-step instructions.

Problem: Non-continuous classroom numbering

32 participants identified discontinuous room numbering as a major source of confusion, making it the most common pain point. The strongest mechanism is guided route breakdown instead of expecting users to infer the layout themselves. This suggests students need direct instructions rather than spatial guesswork. As a result, the design should prioritize concise action-based guidance and a simple top-down route view.

Problem: Losing direction during indoor-outdoor transitions

24 participants said they lost orientation when moving between outside and inside spaces. The strongest mechanism is point-of-entry orientation through an immediately visible electronic sign. This is reinforced by 24 participants saying they would use the system at first sight in an unfamiliar place, while 21 said they would use it after spending too long searching. This supports placing the system at building entrances and designing it for instant use under time pressure.

The severity of the problem was also clear. Only 8 participants found a classroom within five minutes in their worst experience, 16 took 5 to 10 minutes, 17 spent more than 10 minutes, and 9 skipped class because they could not find the room. In addition, 38 participants had been late because of classroom-finding difficulties. These findings show that the system must reduce hesitation, save time, and restore confidence at key decision points.

3. Requirement Analysis

A stacked bar chart helps transform questionnaire findings into concrete design requirements by mapping each pain point to the navigation mechanism most needed to address it. This makes the analysis meaningful not only descriptively, but also as a design translation tool that connects observed frustrations to specific interface features.

Requirement Focus: Structured destination search

The strongest corresponding feature is structured destination search. This indicates that students need the system to interpret room targets more clearly than static signs do. This supports searchable room input, building-aware matching, and reduced ambiguity in destination selection.

Requirement Focus: Floor-specific route guidance

The strongest corresponding feature is floor-specific route guidance. This reflects that students need explicit confirmation of vertical movement, not just a final room number. Therefore, the system should clearly communicate floor transitions as part of the route.

Requirement Focus: Step-by-step route instruction

The strongest feature is step-by-step route instruction supported by a simplified visual path. This indicates that students need direct navigation help instead of relying on room-number patterns that seem inconsistent. As a result, the interface should lower cognitive load through short action-based instructions and a legible route display.

Requirement Focus: Entrance-based orientation

The strongest feature is entrance-based orientation. This suggests that students need support at the moment they enter the building, when uncertainty is highest. This supports a requirement for a point-of-entry electronic sign, a clear "you are here" starting point, and immediate route generation from the chosen entrance.

These mechanisms are not decorative additions. They are direct responses to observed user problems. At a system level, this means the final design must be quick to access, easy to understand, building-specific, and capable of guiding users from uncertainty to confident movement with minimal effort.

3.2 Design Alternatives

Alternative Strengths Limitations Decision
2D Top-Down Map + Text Steps Fast scanning, low visual noise, easy to explain in kiosk context. Less spatial realism than 3D view. Chosen (better first-time task completion).
3D Isometric Navigation View Stronger realism and visual attractiveness. Higher cognitive load and slower decision speed for some users. Not selected for alpha version.
Text-Only Direction List Implementation is simple and lightweight. No visual orientation support; hard at complex intersections. Kept only as supportive layer, not primary mode.

Why 2D Top-Down Map + Text Steps was chosen

Overall evaluation: This option offered the strongest balance of clarity, speed, and first-time usability for a public kiosk wayfinding context.

Compared to the 3D isometric view, it retained clearer route readability and introduced less cognitive load during fast decision-making. Compared to the text-only list, it offered more visual orientation support while remaining simpler and easier to interpret than a more spatially realistic interface. For these reasons, it was chosen as the most effective solution for helping students complete classroom-finding tasks quickly and confidently.

ROUTING LOGIC

Route Generation Logic

The routing logic translates the static EB building graph into user-facing guidance. It does not require live GPS tracking; instead, it uses the selected entrance and classroom code to calculate a predictable indoor route.

1. Start and destination mapping

The selected entrance maps to a start node, while the room code maps to a destination node or room position.

2. Graph-based route lookup

The app searches connected route data to find a walkable path through corridors, stairs, and floor transitions.

3. Floor-aware route state

The route output is grouped by floor so users can understand when they need to move between levels.

4. UI rendering

The map highlights the route by floor, shows instruction cards, and marks the target room.

This architecture fits a lightweight browser-based kiosk because route data is static, predictable, and fast to run locally in the frontend.

C. Key Alpha Insights

1

Users understood the room-code search idea, but needed the input area to be more visually dominant.

2

Users relied heavily on text steps at stairs and corridor turns.

3

Users wanted clearer confirmation that they had reached the correct destination.

4

Floor changes needed to be separated into explicit route steps.

5.2 Alpha Feedback → Design Refinement

The alpha test feedback directly informed the interface refinement shown below.

Feedback → Change → Result

Alpha Feedback Design Change Result After Refinement
Users were unsure where to enter the classroom code. Added a more visible input box for direct room-code entry. Users could start the task faster with less hesitation.
Users relied on text guidance at stairs and corridor turns. Added clearer step-by-step instruction cards. Route interpretation became easier at decision points.
Users were uncertain during floor transitions. Separated stair and floor-change actions into explicit route steps. Users understood when and how to move between floors.
Users wanted confirmation after reaching the target room. Added a clearer arrival/completion state. Users felt more confident that the route was complete.

Before: High Cognitive Load

  • Users had to select their target classroom only through the touchscreen panel.
  • This interaction was slower when users already knew the room code.
  • First-time users could feel uncertain while browsing for the correct classroom.

After: Active Guidance

  • An input box was added so users could enter the classroom code directly.
  • This made destination selection faster and more efficient for time-sensitive users.
  • Clearer input flow improved confidence and reduced unnecessary interaction steps.

5.4 Final Reflection

Societal Impact

  • CampusCompass helps new students, visitors, and occasional users navigate unfamiliar EB building spaces more confidently.
  • By reducing hesitation and wrong turns, the system supports punctuality and lowers wayfinding stress.
  • The project promotes more independent movement through the building, especially for first-time users.

Ethical Considerations

  • The current prototype does not track users or store personal location data.
  • Research photos and feedback records should be used with participant consent or anonymisation.
  • Future versions should better support accessibility needs, including wheelchair routes and elevator-first navigation.

AI Use Reflection

  • AI tools supported wording refinement, coding assistance, debugging, and portfolio structure.
  • The team reviewed AI-assisted suggestions before using them in the final project.
  • Final research interpretation, design decisions, user testing, and ethical judgement remained the responsibility of the team.
