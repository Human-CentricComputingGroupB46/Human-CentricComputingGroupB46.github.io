CampusCompass

Room codes show the floor. We show the way.

A browser-based EB building navigation kiosk that turns confusing room codes into entrance-aware routes for new students and visiting guests.

Project Overview

CampusCompass addresses a common EB building problem: room codes such as EB204 reveal the floor but not the wing, closest entrance, or turning decisions. Users choose their entrance, enter a target room, and receive map-based guidance with concise steps.

The Why

Our group selected the Active track because indoor navigation is a real, frequent friction point in everyday campus movement. In EB, users may know the room code but still fail to locate the room efficiently because the code does not reveal wing-level direction. This creates repeated stopping, backtracking, and uncertainty, especially for first-year students in their first weeks and external guests who visit only once. We observed that this problem is not caused by a lack of motivation to navigate, but by a mismatch between static signage and the moment-to-moment decisions users need to make while moving. We therefore designed CampusCompass as a point-of-entry decision support tool: users declare where they entered, enter a destination code, and immediately receive a shortest path and plain-language steps. This directly supports quick, confident movement under time pressure and lowers cognitive load. The concept is intentionally lightweight, browser-based, and deployable on a kiosk screen, so it can be installed where confusion starts rather than requiring every user to install a new app. The project reflects human-centered priorities: reducing anxiety, improving first-time success, and enabling independent navigation for a broader range of users.

The Gap: Academic Solutions

  • Did well: clear shortest-path logic and graph-based modelling.
  • Did well: strong technical metrics for accuracy and computation speed.
  • Did well: multi-floor route handling through stairs/lifts.
  • Missed: weak attention to first-time-user anxiety and confidence cues.
  • Missed: limited communication design for non-technical users.
  • Missed: poor transfer from algorithm output to practical on-site decisions.

The Gap: Commercial Products

  • Did well: polished UI and quick visual orientation.
  • Did well: robust map rendering and broad place coverage.
  • Did well: familiar interaction patterns and low learning cost.
  • Missed: often require heavy setup, network, or app installation.
  • Missed: weak custom fit for one confusing building context.
  • Missed: less transparent route reasoning for stressed first-time visitors.

Design Alternatives Comparison

Alternative Strengths Limitations Decision
2D Top-Down Map + Text Steps Fast scanning, low visual noise, easy to explain in kiosk context. Less spatial realism than 3D view. Chosen (better first-time task completion).
3D Isometric Navigation View Stronger realism and visual attractiveness. Higher cognitive load and slower decision speed for some users. Not selected for alpha version.
Text-Only Direction List Implementation is simple and lightweight. No visual orientation support; hard at complex intersections. Kept only as supportive layer, not primary mode.

5. Did Users Find the Way?

Alpha Findings

Usability Testing (Alpha)

  • Participants: 3 real users.
  • Task success: 100%.
  • Average time saved: around 60% vs unaided navigation.
  • Interaction efficiency: users reached route view within 3 taps.

Iterative Refinement

  • Before: route information presentation had higher cognitive load for first-time users.
  • After: final alpha version uses 2D top-down map + concise step list + completion cue.
  • Impact: faster understanding, fewer hesitations at decision points.

Material slot: before/after screenshots.

Before Alpha Refinement

  • Route information felt visually dense for first-time users.
  • Users still had to mentally translate map output into next actions.
  • Completion feedback was not prominent enough.

After Alpha Refinement

  • 2D top-down route display became the primary guidance layer.
  • Concise step list reduced memory load at decision points.
  • Clear route and completion cues made users feel more confident.

Final Reflection (Social / Ethical / AI)

CampusCompass aims to improve inclusion by reducing stress for new students and external visitors who lack spatial familiarity. The system is privacy-preserving in its current form because it runs fully client-side without collecting personal data. For future deployment, any analytics should remain aggregated and clearly disclosed. In this project process, AI was used as an assistant for ideation and communication support, while core problem framing, design decisions, implementation logic, and evaluation interpretation remained team-driven.
